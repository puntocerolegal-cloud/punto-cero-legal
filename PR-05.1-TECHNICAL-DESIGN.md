# PR-05.1 — Error Handling con Valor Real

## 1. Problema Concreto

### Estado Actual (Sin Manejo de Errores)

Los servicios de OS actualmente **no capturan ni procesan errores HTTP explícitamente**:

```js
// billing.service.js — createInvoice()
async createInvoice(payload) {
  const data = normalizeInvoice(unwrap(await apiClient.post("/billing/", payload)));
  eventBus.emit(OS_EVENTS.invoiceCreated, data);
  return data;
}
```

Si `apiClient.post()` falla (400, 401, 403, 404, 409, 422, 500, timeout, CORS), la promesa rechazada **propaga sin contexto**:
- No se sabe qué operación falló (¿crear? ¿actualizar?)
- No se sabe sobre qué recurso (¿qué factura? ¿qué suscripción?)
- No se sabe qué tipo de error HTTP es
- El error es un objeto Axios crudo, no normalizado
- No hay registro centralizado

### Impacto en Usuarios Finales
- UI muestra mensajes genéricos o crashes silenciosos
- No puede distinguir errores recuperables (409 conflict) de no-recuperables (500)
- Sin reintentos inteligentes, sin fallbacks, sin acciones contextuales

### Impacto en Desarrolladores
- Difícil debuggear: el stack trace no dice qué operación falló
- Sin patrón uniforme: cada consumidor del servicio maneja errores diferente
- Código defensivo duplicado en 20+ módulos que usan estos servicios

### Impacto en Observabilidad
- Sin logs estructurados: logs de consola son strings crudo
- Sin trazabilidad: no se sabe cuántos usuarios afectados
- Sin métricas: no se mide qué endpoints fallan más

### Diseño de `frontend/src/lib/osErrorHandler.js`

```js
/**
 * Clasificación de errores HTTP para tomar decisiones en UI/lógica.
 */
export const ErrorKind = {
  NOT_FOUND: 'not_found',           // 404: recurso no existe
  CONFLICT: 'conflict',             // 409: state conflict (ej: estado esperado != actual)
  VALIDATION: 'validation',         // 422: datos inválidos
  UNAUTHORIZED: 'unauthorized',     // 401: token inválido/expirado
  FORBIDDEN: 'forbidden',           // 403: sin permiso
  SERVER_ERROR: 'server_error',     // 5xx: error interno del backend
  NETWORK_ERROR: 'network_error',   // Red caída, timeout, CORS
};

/**
 * Error de dominio: normaliza cualquier error HTTP en estructura consistente.
 * Los consumidores reciben siempre esta forma, nunca Axios crudo.
 */
export class OSError extends Error {
  constructor({
    kind,                // ErrorKind
    statusCode,          // 404, 409, 422, 500, etc.
    service,             // 'billing', 'subscriptions', 'partners'
    operation,           // 'createInvoice', 'updateSubscription'
    resourceId,          // id del recurso afectado (opcional)
    resourceType,        // 'invoice', 'subscription', 'partner' (opcional)
    userMessage,         // Mensaje para mostrar en UI (español)
    details,             // Objeto con { detail, errors, field, ... }
    originalError,       // El error Axios original
  } = {}) {
    super(userMessage || `Error en ${service}.${operation}`);
    this.name = 'OSError';
    this.kind = kind;
    this.statusCode = statusCode;
    this.service = service;
    this.operation = operation;
    this.resourceId = resourceId;
    this.resourceType = resourceType;
    this.details = details;
    this.originalError = originalError;
  }

  // Checks para lógica de UI
  isNotFound() { return this.kind === ErrorKind.NOT_FOUND; }
  isConflict() { return this.kind === ErrorKind.CONFLICT; }
  isValidation() { return this.kind === ErrorKind.VALIDATION; }
  isUnauthorized() { return this.kind === ErrorKind.UNAUTHORIZED; }
  isForbidden() { return this.kind === ErrorKind.FORBIDDEN; }
  isServerError() { return this.kind === ErrorKind.SERVER_ERROR; }
  isNetworkError() { return this.kind === ErrorKind.NETWORK_ERROR; }

  // Conversión a log estructurado
  toLog() {
    return {
      timestamp: new Date().toISOString(),
      error: this.message,
      kind: this.kind,
      statusCode: this.statusCode,
      service: this.service,
      operation: this.operation,
      resourceId: this.resourceId,
      resourceType: this.resourceType,
      details: this.details,
    };
  }
}

/**
 * Normalizador universal: convierte Axios error → OSError.
 * Enriquece con contexto de la operación que falló.
 */
export function normalizeHTTPError(
  axiosError,
  { service, operation, resourceId, resourceType } = {}
) {
  const statusCode = axiosError?.response?.status;
  const detail = axiosError?.response?.data?.detail;

  let kind = ErrorKind.NETWORK_ERROR;
  let userMessage = 'No se pudo conectar con el servidor.';

  if (statusCode === 404) {
    kind = ErrorKind.NOT_FOUND;
    userMessage = `${resourceType || 'El recurso'} no existe.`;
  } else if (statusCode === 409) {
    kind = ErrorKind.CONFLICT;
    userMessage = 'El estado ha cambiado. Recargue e intente de nuevo.';
  } else if (statusCode === 422) {
    kind = ErrorKind.VALIDATION;
    userMessage = detail || 'Los datos no son válidos.';
  } else if (statusCode === 401) {
    kind = ErrorKind.UNAUTHORIZED;
    userMessage = 'Su sesión ha expirado. Inicie sesión de nuevo.';
  } else if (statusCode === 403) {
    kind = ErrorKind.FORBIDDEN;
    userMessage = 'No tiene permiso para hacer esto.';
  } else if (statusCode >= 500) {
    kind = ErrorKind.SERVER_ERROR;
    userMessage = 'Error del servidor. Intente de nuevo en unos momentos.';
  }

  return new OSError({
    kind,
    statusCode,
    service,
    operation,
    resourceId,
    resourceType,
    userMessage,
    details: { detail, ...axiosError?.response?.data },
    originalError: axiosError,
  });
}
```

### Uso en Servicios: Ejemplo `billing.service.js`

**ANTES:**
```js
async createInvoice(payload) {
  const data = normalizeInvoice(unwrap(await apiClient.post("/billing/", payload)));
  eventBus.emit(OS_EVENTS.invoiceCreated, data);
  return data;
}
```

**DESPUÉS:**
```js
import { normalizeHTTPError } from "@/lib/osErrorHandler";

async createInvoice(payload) {
  try {
    const data = normalizeInvoice(unwrap(await apiClient.post("/billing/", payload)));
    eventBus.emit(OS_EVENTS.invoiceCreated, data);
    return data;
  } catch (err) {
    throw normalizeHTTPError(err, {
      service: 'billing',
      operation: 'createInvoice',
      resourceType: 'invoice',
    });
  }
}
```

## 4. Cambio Medible

### Para el Usuario
- Errores más específicos en UI: "No existe" vs "Sin permiso" vs "Datos inválidos"
- Posibilidad de reintentos inteligentes según el tipo de error
- Mejor experiencia de error: mensajes claros en español

### Para el Desarrollador
- Logs estructurados que dicen: `{ service: 'billing', operation: 'createInvoice', resourceId: 'INV-123', kind: 'conflict' }`
- Posibilidad de manejar errores específicos:
  ```js
  try {
    await billingService.createInvoice(payload);
  } catch (err) {
    if (err.isConflict()) {
      // Reintentar con datos frescos
    } else if (err.isValidation()) {
      // Mostrar errores de validación
    }
  }
  ```

### Para Observabilidad
- Métricas: cuántos 404 vs 409 vs 500 por servicio
- Alertas: si un endpoint tiene >10% de errores 500
- Trazabilidad: qué operaciones fallan más, en qué recursos

### Para Estabilidad
- Cambio de comportamiento: **cero**. Los errores aún se lanzan.
- Cambio de API: los consumidores reciben `OSError` en lugar de `AxiosError`.
- Beneficio: estructura consistente permite patrones defensivos uniformes.

## 5. Ejemplo: Antes y Después

### Error de Conflicto (409) — Actualizar Suscripción con Estado Viejo

#### ANTES:
```
AxiosError {
  message: 'Request failed with status code 409',
  code: 'ERR_BAD_REQUEST',
  response: {
    status: 409,
    data: { detail: 'La suscripción cambió de estado' }
  },
  config: { url: '/subscriptions/SUB-123', method: 'put', ... },
  request: XMLHttpRequest,
  ...
}
```

**Consumidor no sabe:**
- Qué operación falló exactamente
- Si el recurso existe o no
- Si debe reintentar o no

#### DESPUÉS:
```
OSError {
  message: 'El estado ha cambiado. Recargue e intente de nuevo.',
  kind: 'conflict',
  statusCode: 409,
  service: 'subscriptions',
  operation: 'update',
  resourceId: 'SUB-123',
  resourceType: 'subscription',
  toLog() {
    {
      timestamp: '2026-06-30T14:22:33.456Z',
      error: 'El estado ha cambiado. Recargue e intente de nuevo.',
      kind: 'conflict',
      statusCode: 409,
      service: 'subscriptions',
      operation: 'update',
      resourceId: 'SUB-123',
      resourceType: 'subscription',
    }
  }
}
```

**Consumidor puede:**
```js
try {
  await subscriptionsService.update('SUB-123', newPayload);
} catch (err) {
  if (err.isConflict()) {
    // Refrescar datos y reintentar
    const fresh = await subscriptionsService.getDetails('SUB-123');
    // Mostrar: "Datos actualizados, intente de nuevo"
  } else if (err.isValidation()) {
    // Mostrar errores de campo
    toast.error(err.message);
  }
}
```

## 6. Impacto en Producción

### Riesgo
- **Muy Bajo**: solo servicios internos (17 métodos), comportamiento de error no cambia.
- **Mitigation**: todos los errores son relanzados; no se silencian.
- **Testeo**: se prueba el mapping de errores sin tocar UI.

### Rollback
- Remover try/catch de los 15 servicios (5 minutos).
- Los servicios volverían a lanzar `AxiosError` crudo.
- Sin dependencias externas o cambios de datos.

### Beneficio Acumulado
- Punto 1: patrón uniforme para 40 métodos de mutación.
- Punto 2: base para reintentos inteligentes, auditoría, y observabilidad futura.
- Punto 3: todos los servicios quedan "listo para fallar gracefully".
