# PR-05.2 — Auditoría Técnica de osErrorHandler.js

## 1. Resumen Ejecutivo

El módulo `osErrorHandler.js` es un archivo de normalización de errores HTTP diseñado para enriquecer excepciones de Axios con contexto de dominio. El análisis revela que el módulo **cumple estrictamente con el principio de responsabilidad única** y mantiene un enfoque coherente y bien delimitado.

**Veredicto Preliminar:** Módulo técnicamente sólido, sin sobreingeniería, sin responsabilidades mezcladas.

## 4. Hallazgos

### Hallazgo 1: Single Responsibility Principle ✓
**Evidencia:**
- El módulo tiene UNA responsabilidad clara: normalizar errores HTTP
- Cada función/clase tiene UNA tarea:
  - `ErrorKind`: Define categorías
  - `OSError`: Encapsula error normalizado
  - `normalizeHTTPError()`: Convierte Axios → OSError

**Conclusión:** Cumple SRP perfectamente. No hay responsabilidades mezcladas.

---

### Hallazgo 2: Acoplamiento Bajo ✓
**Dependencias Externas:**
- `Error` (clase nativa de JavaScript)
- `Object.entries()` (método nativo)
- Nada más

**Dependencias Internas:**
- Ninguna (no importa otros módulos)

**Conclusión:** Acoplamiento cero con el proyecto. Pure module.

---

### Hallazgo 3: Cohesión Alta ✓
**Evidencia:**
- `ErrorKind` se usa internamente en `normalizeHTTPError()` y `OSError`
- `OSError` se retorna por `normalizeHTTPError()`
- Los métodos `is*()` de OSError usan la propiedad `kind` interna
- Todo está relacionado conceptualmente: clasificación → error → queries

**Conclusión:** Alta cohesión. Las partes están relacionadas y se apoyan mutuamente.

### Hallazgo 4: Complejidad Ciclomática
**Análisis:**

  1 (entrada)

  + 1 (statusCode === 409?)
  + 1 (statusCode === 422?)
  + 1 (statusCode === 401?)
  + 1 (statusCode === 403?)

**Métodos is*():**
- Cada uno tiene ciclomática = 1 (trivial)

**Constructor de OSError:**
- Ciclomática ≈ 1-2 (trivial, solo asignaciones condicionales)

**Conclusión:** Complejidad baja a moderada. Fácilmente testeable.

### Hallazgo 5: Flujo de Ejecución
**Entrada:** `normalizeHTTPError(axiosError, context)`  
**Proceso:**
1. Extrae `statusCode`, `detail`, `responseData` del error
2. Inicializa `kind = NETWORK_ERROR` (default)
3. Aplica 6 clasificaciones por statusCode
4. Construye `OSError` con contexto enriquecido
5. **Siempre lanza** (`throw osError`) — nunca silencia

**Salida:** Lanza `OSError` enriquecido o muere (no devuelve `undefined`)

**Conclusión:** Flujo simple, predecible, determinístico.

### Hallazgo 7: Clasificación de Errores HTTP
**Categorías Implementadas:**
- 404 (NOT_FOUND)
- 409 (CONFLICT)
- 422 (VALIDATION)
- 401 (UNAUTHORIZED)
- 403 (FORBIDDEN)
- 5xx (SERVER_ERROR)
- Default (NETWORK_ERROR) — timeouts, CORS, sin respuesta

**Cobertura:** Las categorías más comunes en APIs REST están cubiertas.

**No Implementadas (justificado):**
- 429 (RATE_LIMIT) — no es crítico inicialmente
- 503 (SERVICE_UNAVAILABLE) — caería en NETWORK_ERROR (similar)
- Otros 4xx — no son tan comunes (caen en NETWORK_ERROR)

**Conclusión:** Suficiente para MVP. Extensible si es necesario.

### Hallazgo 8: Mensajes de Usuario
**Evidencia (líneas 127-145):**
```js
if (statusCode === 404) {
  userMessage = detail || `${resourceType || 'El recurso'} no existe.`;
}
```

**Análisis:**
- Usa `detail` del backend si existe
- Fallback a mensaje generic con `resourceType`
- Todos en español (hardcoded)
- Amigables para usuario final

**Nota:** Los mensajes no están extraídos a constantes i18n, pero eso está fuera del scope actual (no es sobreingeniería, es apropiado para MVP).

**Conclusión:** Mensajes adecuados, sin exceso de complejidad.

---

### Hallazgo 9: Métodos Helper is*()
**Evidencia (líneas 72-95):**
```js
isNotFound() { return this.kind === ErrorKind.NOT_FOUND; }
isConflict() { return this.kind === ErrorKind.CONFLICT; }
...
```

**Análisis:**
- 7 métodos boilerplate simples
- Cada uno hace UNA comparación
- Evita que consumidores inspeccionen `statusCode` directamente
- Costo: trivial (lecturas de propiedad)

**Pregunta:** ¿Es sobreingeniería tener 7 métodos para 7 comparaciones?

**Respuesta:** NO.
- Razón 1: Facilita queries sin conocer ErrorKind enum
- Razón 2: Si ErrorKind cambia estructura, los métodos lo abstraen
- Razón 3: Patrón estándar en classes de error (similar a Error.prototype)
- Razón 4: No hay alternativa mejor (no puedes hacer `err.is[kind]()`)

**Conclusión:** Inversión mínima, retorno máximo. Decisión correcta.

## 5. Riesgos

### Riesgo 1: Mensajes Hardcoded en Español 🟡
**Descripción:**
Los mensajes `userMessage` están en español hardcoded:
```js
userMessage = 'No se pudo conectar con el servidor.';
userMessage = detail || `${resourceType || 'El recurso'} no existe.`;
```

**Impacto:** Si el proyecto necesita i18n en el futuro, estos strings será difícil encontrar y traducir.

**Probabilidad:** Media (es posible pero no inmediato)

**Severidad:** Baja (es fácil extraer cuando sea necesario)

**Clasificación:** 🟡 Medio (bajo-medio)

---

### Riesgo 2: statusCode === vs >= (Inconsistencia Sintáctica)
**Descripción:**
La mayoría usa `===` pero la última categorización usa `>=`:
```js
if (statusCode === 404) { ... }
if (statusCode === 409) { ... }
...
} else if (statusCode && statusCode >= 500) {  // <= acá usa >=
```

**Impacto:** Código legible pero con pequeña inconsistencia. No afecta comportamiento.

**Severidad:** Muy baja (es intencional y correcto para agrupar 5xx)

**Clasificación:** 🟢 Bajo (apenas notable)

---

### Riesgo 3: Sin Validación del Parámetro axiosError
**Descripción:**
`normalizeHTTPError()` no valida que `axiosError` sea efectivamente un Error:
```js
export function normalizeHTTPError(axiosError, context = {}) {
  const statusCode = axiosError?.response?.status;  // Si axiosError es null, OK (?.=
```

**Impacto:** Si se llama con `normalizeHTTPError(null, ...)`, devuelve un OSError válido con valores por defecto. No rompe, solo pierde contexto.

**Severidad:** Baja (el optional chaining (`?.`) lo maneja gracefully)

**Clasificación:** 🟢 Bajo (es defensivo, no vulnerable)

---

### Riesgo 4: Sin Manejo de errorArray (Validación Pydantic)
**Descripción:**
El backend puede devolver `detail` como array de errores (Pydantic v2):
```js
response.data = {
  detail: [
    { loc: ['body', 'email'], type: 'value_error', msg: '...' }
  ]
}
```

El módulo lo mantiene tal cual en `details.detail`, pero `userMessage` no lo procesa:
```js
userMessage = detail || 'Los datos no son válidos.';
```

Si `detail` es array, `userMessage` será `[Object Object]`.

**Impacto:** La UI podría mostrar mensajes de error poco legibles para validaciones Pydantic.

**Severidad:** Baja (hay fallback genérico, y hay otro lugar en el proyecto que sí lo maneja: `getErrorMessage()` en `utils.js`)

**Clasificación:** 🟡 Medio (mejora futura, no crítica hoy)
