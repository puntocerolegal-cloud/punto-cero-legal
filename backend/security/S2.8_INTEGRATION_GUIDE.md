S2.8 transforms Punto Cero Legal into a **Self-Defending Security Operating System**.

**Request Flow:**
```
Request
  ↓

  ↓
```

## 🧠 Core Components

### 1. **Autonomous Decision Engine** (`autonomous_decision_engine.py`)

**Purpose:** Convert risk signals into automated decisions.

**Decision Matrix:**
- Risk 0-39: `ALLOW` — Normal operation
- Risk 40-59: `MONITOR` — Enhanced logging
- Risk 60-74: `THROTTLE` — Rate limiting
- Risk 75-84: `BLOCK` — Deny access
- Risk 85-94: `ISOLATE_USER` — Sandbox user
- Risk 95-100: `ISOLATE_TENANT` + `FAIL_SAFE` — Full isolation + system lockdown

**Integration Points:**
```python
from backend.security.autonomous_decision_engine import get_autonomous_decision_engine

decision_engine = get_autonomous_decision_engine()
decision = decision_engine.decide(
    user_id=user_id,
    tenant_id=tenant_id,
    risk_score=risk_score,
    attack_graph_state=attack_graph_state,
    behavioral_deviation=behavioral_deviation,
    tenant_risk=tenant_risk,
    correlation_signals=correlation_signals,
)
# decision.decision_type → DecisionType enum
# decision.actions → list of mitigation actions
# decision.confidence → 0-1 confidence score
```

---

### 2. **Mitigation Engine** (`mitigation_engine.py`)

**Purpose:** Execute autonomous response actions in real-time.

**Actions:**
- `rate_limit_user(user_id, requests_per_minute)`
- `block_user(user_id, duration_seconds)`
- `block_ip(ip, duration_seconds)`
- `quarantine_tenant(tenant_id, mode="read_only")`
- `revoke_tokens(user_id)`
- `reduce_permissions(user_id, restrictions)`
- `isolate_user(user_id, mode="sandbox")`
- `trigger_fail_safe(scope, target)`

**Integration Points:**
```python
from backend.security.mitigation_engine import get_mitigation_engine

mitigation_engine = get_mitigation_engine()
actions = [
    {"type": "rate_limit", "target": user_id, "requests_per_minute": 10},
    {"type": "increase_audit_logging", "target": user_id, "intensity": "critical"},
]
results = await mitigation_engine.execute_actions(actions)
```

---

### 3. **Auto-Policy Engine** (`auto_policy_engine.py`)

**Purpose:** Dynamically adjust security policies at runtime.

**Policy Modes:**
- `NORMAL` — Standard restrictions
- `ELEVATED` — No deletes, MFA required
- `STRICT` — Read-only, MFA required, IP whitelist
- `LOCKDOWN` — Read-only, 1 connection, IP whitelist, audit all

**Integration Points:**
```python
from backend.security.auto_policy_engine import get_auto_policy_engine

policy_engine = get_auto_policy_engine()

# Adjust tenant policy
policy = policy_engine.adjust_tenant_policy(
    tenant_id=tenant_id,
    new_mode=PolicyMode.STRICT,
    reason="Repeated attacks detected",
    duration_seconds=7200,
)

# Check if operation allowed
allowed = policy_engine.is_operation_allowed(
    tenant_id=tenant_id,
    operation_type="delete",
    resource_type="document",
)
```

---

### 4. **Containment Engine** (`containment_engine.py`)

**Purpose:** Isolate threats without affecting entire system.

**Containment Scopes:**
- `USER` → Sandbox individual user
- `TENANT` → Quarantine entire tenant (read-only)
- `ENDPOINT` → Disable specific endpoint
- `SESSION` → Sandbox specific session

**Containment Modes:**
- `MONITORED` — Log all activity
- `SANDBOX` — Fake responses, redirect DB access
- `READONLY` — No writes/deletes
- `DISABLED` — Return 503 Service Unavailable

**Integration Points:**
```python
from backend.security.containment_engine import (
    get_containment_engine,
    ContainmentScope,
)

containment = get_containment_engine()

# Isolate user
record = await containment.isolate_user_scope(
    user_id=user_id,
    reason="Attack detected",
    duration_seconds=3600,
)

# Check containment status
is_contained = containment.is_user_contained(user_id)
status = containment.get_containment_status(user_id, ContainmentScope.USER)
```

---

### 5. **Recovery Engine** (`recovery_engine.py`)

**Purpose:** Restore normalcy after incident containment.

**Recovery States:**
- `ACTIVE_THREAT` — Threat ongoing
- `THREAT_CONTAINED` → `SAFE_RECOVERY` — Begin recovery
- `GRADUAL_RESTORE` → Permissions restored in stages
- `FULL_RESTORE` — Complete recovery

**Integration Points:**
```python
from backend.security.recovery_engine import get_recovery_engine

recovery = get_recovery_engine()

# Initiate recovery
plan = await recovery.initiate_recovery(
    target_id=user_id,
    target_type="user",
    attack_duration_seconds=3600,
)

# Monitor recovery
progress = recovery.get_recovery_progress(user_id)
# → {"state": "gradual_restore", "progress_percent": 33.0, ...}

# Gradual permission restoration
result = await recovery.gradually_restore_permissions(
    target_id=user_id,
    original_permissions=original_perms,
)
```

## 🔌 Integration into Security Engine

### In `security_engine.py`

**S2.8 is integrated as a "response hook" that fires on high-risk decisions:**

```python
async def authorize(...):
    # ... existing S2.5 authorization logic ...
    
    # S2.8: Apply autonomous response if risk context provided
    if context and context.get("risk_context"):
        await _apply_autonomous_response(
            user, resource_type, action, resource, context["risk_context"]
        )
```

**Usage in endpoints:**
```python
from backend.routes import auth

@app.get("/api/cases/{case_id}")
async def get_case(
    case_id: str,
    current_user: dict = Depends(auth.get_current_user),
    db: AsyncIOMotorDatabase = Depends(auth.get_db),
):
    # Fetch case with risk context
    case = await db.cases.find_one({"_id": ObjectId(case_id)})
    
    # Compute risk context (from S2.6)
    risk_context = {
        "risk_score": 65,
        "attack_graph_state": {...},
        "behavioral_deviation": 45,
        "tenant_risk": 30,
        "correlation_signals": [...],
    }
    
    # Authorize with S2.8 response hook
    await authorize(
        user=current_user,
        resource_type="case",
        action="read",
        resource=case,
        context={"risk_context": risk_context},
        db=db,
    )
    
    return case
```

## 🚀 Operational Modes

### Mode 1: Passive Monitoring (Default)
- S2.8 enabled but only logs recommendations
- No actions executed
- Used in initial deployment

**Configuration:**
```python
class AutonomousDecisionEngine:
    EXECUTION_MODE = "audit_only"  # "audit_only" | "full"
```

### Mode 2: Automatic Mitigation (Active)
- S2.8 executes decisions automatically
- High-confidence decisions bypass human approval
- Critical actions notify SOC immediately

**Configuration:**
```python
class AutonomousDecisionEngine:
    EXECUTION_MODE = "full"
    APPROVAL_THRESHOLD = 0.85  # Confidence level for auto-execution
```

### Mode 3: Fail-Safe (Emergency)
- S2.8 triggers on any critical threat
- System enters read-only mode
- All writes blocked, only reads allowed
- SOC alerted immediately

**Configuration:**
```python
class AutonomousDecisionEngine:
    FAILSAFE_ENABLED = True
    FAILSAFE_RISK_THRESHOLD = 95  # Trigger at risk > 95
```

## 🔄 Autonomous Loop Cycle

**Continuous system operation:**

```
Every Request:
  1. Detect (S2.6)
     ├─ Behavioral analysis
     ├─ Attack graph evaluation
     ├─ Risk computation
     └─ Threat correlation

  2. Decide (S2.8)
     ├─ Risk → Decision mapping
     ├─ Confidence calculation
     └─ Action selection

  3. Act (Mitigation + Containment + Policy)
     ├─ Execute immediate actions
     ├─ Adjust policies
     └─ Isolate threats

  4. Observe (S2.7)
     ├─ Log all autonomous actions
     ├─ Update incident status
     └─ Stream to SOC

  5. Learn (S2.6 Feedback Loop)
     ├─ Track decision accuracy
     ├─ Adjust thresholds
     └─ Update profiles
```

## ⚠️ Critical Safety Rules

✅ **ALWAYS:**
- Log all autonomous actions to S2.7 SOC
- Check confidence score before high-impact actions
- Preserve data integrity (never delete, only isolate)
- Allow manual override by SOC operators
- Graceful degradation to read-only on failure

❌ **NEVER:**
- Execute actions without risk context
- Skip audit logging
- Make irreversible changes without backup
- Trigger FAIL_SAFE without critical threshold
- Block system admins or SOC operators
